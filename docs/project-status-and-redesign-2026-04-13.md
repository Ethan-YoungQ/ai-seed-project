## 二、三个线程的核心发现

### 线程 A：卡片回调调试（本线程上一轮）

**结论**：WS 长连接的卡片回调可工作，但只能返回 toast，不能返回 card 更新。
Codex 建议未来用异步 `im.message.patch` API 刷新卡片。

**经验沉淀**：
- 详见 `memory/project_feishu_ws_card_action_lessons.md`
- 详见 `skill: feishu-card-ws`

### 线程 B：系统功能断点分析

**核心发现**：17 种卡片中，只有管理面板接通了触发，其余 5 种 P0 功能卡片"断路"——
模板和 Handler 都写好了，但没有人把卡片发到群里。

**原始方案**：为签到/测验/作业/视频/互评各加一个关键词触发。

**但用户明确指出**：这个方案"交互过重"——所有学员操作都依赖飞书卡片是反人性的。

### 线程 C：服务器安全与公网入口

**核心发现**：
- 当前杭州裸 IP + HTTP 3000 端口 **不适合正式上线**
- 无 HTTPS、无域名、可能被查封
- 推荐方案：阿里云中国香港 SWAS + 域名 + HTTPS
- 或者：收敛到飞书原生入口，放弃独立公网看板

## 三、用户的重新设计方向

### 3.1 核心思想

> "废除卡片设计的过重。所有的作业提交如果都依赖于飞书卡片，这显然是反人性的。
> 管理员的操作可以依赖于卡片，其他都应该改成 bot 自动抓取、自动 LLM 评分、
> 自动进入 Dashboard。"

### 3.2 当前设计 vs 目标设计

| 维度 | 当前设计（卡片驱动） | 目标设计（自动捕获） |
|------|---------------------|---------------------|
| 学员操作 | 点击卡片按钮提交 | 自然发消息/文件到群里 |
| 触发方式 | 培训师发关键词 → Bot 发卡片 | Bot 自动监听群消息 |
| 评分触发 | 学员在卡片中填写/点击 | Bot 识别内容 → LLM 自动评分 |
| 管理操作 | 管理面板卡片 | **保持不变** |
| 反馈方式 | 卡片更新 / toast | Bot 回复消息确认 |

### 3.3 按评分项分析可行性

| 代码 | 当前方式 | 改造方向 | 可行性 |
|------|---------|---------|--------|
| **K1** 每日打卡 | 点签到卡片按钮 | 学员发"签到"或 Bot 自动识别打卡意图 | ✅ 简单 |
| **K2** 测验答题 | 卡片内答选择题 | **仍需卡片**（结构化选择题无法纯文本替代） | ⚠️ 保留卡片 |
| **K3** 知识总结 | 卡片内填文本 | 学员直接发消息到群，Bot 捕获 → LLM 评 | ✅ 适合自动化 |
| **K4** AI 纠错 | 卡片内填文本 | 学员直接发消息，Bot 识别为纠错内容 | ✅ 适合自动化 |
| **H1** 作业提交 | 卡片点提交按钮 | 学员发文件/截图到群，Bot 自动识别为作业 | ✅ 适合自动化 |
| **H2** 实操分享 | 卡片填文本+截图 | 学员发截图+描述，Bot 捕获 → LLM 多模态评 | ✅ 适合自动化 |
| **H3** 视频打卡 | 卡片点完成按钮 | 学员发消息确认完成，Bot 捕获 | ✅ 简单 |
| **C1** 创意用法 | 卡片填文本 | 学员发创意内容到群，Bot 识别 → LLM 评 | ✅ 适合自动化 |
| **C2** 表情回应 | 给 C1 点 emoji | **已自动**（im.message.reaction 事件） | ✅ 无需改 |
| **C3** 提示词模板 | 卡片填文本 | 学员发 prompt 到群，Bot 识别 → LLM 评 | ✅ 适合自动化 |
| **S1** 互评投票(投) | 卡片投票 | **仍需卡片**（结构化投票需 UI） | ⚠️ 保留卡片 |
| **S2** 互评投票(被投) | 被投票 | 随 S1 自动计算 | ✅ 无需改 |
| **G1** 视频学习 | 卡片点完成 | 学员发消息确认，Bot 捕获 | ✅ 简单 |
| **G2** 课外资源 | 卡片填 URL | 学员发含 URL 的消息，Bot 自动识别 | ✅ 适合自动化 |
| **G3** 全勤加成 | 自动计算 | **已自动** | ✅ 无需改 |

### 3.4 改造结论

**可以完全自动化（不需要卡片）**：K1, K3, K4, H1, H2, H3, C1, C3, G1, G2 — **10 项**
**仍需保留卡片**：K2（选择题测验）、S1（投票） — **2 项**
**已经自动**：C2, S2, G3 — **3 项**

**这意味着 15 项中只有 2 项真正需要卡片交互，其余 13 项都可以通过 Bot 自动捕获群消息实现。**

### 3.5 自动捕获的核心技术方案

```
im.message.receive_v1 事件已通过 WS EventDispatcher 接通
    ↓
onMessage 收到群消息（文本/图片/文件/富文本）
    ↓
意图分类器（简单规则 + LLM 备选）：
  - 包含"签到" → K1 每日打卡
  - 包含 URL → G2 课外资源
  - 包含截图/PDF → H2 实操分享 或 H1 作业提交
  - ≥20字文本 → K3/K4/C1/C3 候选，LLM 进一步分类
  - 包含"完成"/"打卡" → H3/G1 视频打卡
    ↓
Ingestor 10 步管道验证（已完成）
    ↓
需要 LLM 评分的 → LLM Worker 异步处理
    ↓
审核队列 → 管理员审核
    ↓
分数入账 → Dashboard 更新
```

## 五、对 Codex 建议的评估

Codex 建议的"方向 A：异步 patch 链路"在自动捕获方案下**仍然有价值**：
- 管理面板的卡片刷新仍需要 patch
- 审核卡片的状态更新需要 patch
- 但学员侧的交互不再需要卡片 → patch 的优先级降低

**调整后的开发优先级**：
1. **P0**：自动捕获群消息 → 意图分类 → Ingestor（这是新架构的核心）
2. **P0**：服务器迁移方案决定 + 执行
3. **P1**：测验卡片触发接通（唯一必需的卡片）
4. **P1**：互评投票卡片触发接通
5. **P2**：异步 patch 链路（管理面板卡片刷新）
6. **P2**：管理员审核体验优化
