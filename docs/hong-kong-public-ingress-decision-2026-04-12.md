# AI Seed Project 香港公网入口决策文档

日期：2026-04-12  
范围：本文件只讨论“看板公网入口、服务器地域、备案、飞书入口形态、后续建议方案”。  
不在本文件范围内：飞书管理卡片动作回调的具体代码问题。

## 1. 当前主要问题

当前项目已经有一套可运行的后端和 React 看板，但它的公网暴露形态并不适合作为正式上线入口继续扩散。

### 1.1 当前公网入口的事实

本次会话内已确认的对外表现如下：

- `http://114.215.170.79:3000/dashboard/` 返回 `200 OK`
- `http://114.215.170.79/` 返回空响应
- `https://114.215.170.79/` TLS 握手失败

这说明当前真正对外可访问的是“杭州轻量应用服务器上的裸 IP + 3000 端口”，而不是一个已经整理好的正式域名入口。

### 1.2 这为什么是问题

问题不在于“服务能不能打开”，而在于“它是否适合作为正式入口”。

- 现在的入口是裸 IP，而不是域名。
- 现在没有可用的 HTTPS 正式入口。
- 现在的 `dashboard` 依赖同源 API，不是一个独立静态站点。
- 当前一期正式交付口径本来就不是“独立公网 Web 产品”，而是“飞书原生入口 + Feishu Base + 阿里云常驻后端”。

仓库里能直接支持这一判断的证据包括：

- [apps/dashboard/src/lib/api.ts](../apps/dashboard/src/lib/api.ts) 中 `API_BASE = ""`，说明前端默认依赖同源 `/api/...`
- [apps/dashboard/vite.config.ts](../apps/dashboard/vite.config.ts) 中 `base = "/dashboard/"`
- [src/app.ts](../src/app.ts) 里同时托管 `/dashboard/` 和 `/api/*`
- [release-runbook.md](./release-runbook.md) 已明确：一期正式交付面是飞书原生表面与 Base，看板只是工程辅助面

## 2. 我的判断

### 2.1 是否可以迁移到香港服务器

可以，而且这是“保留当前 React 看板，同时尽快形成可用公网入口”的最快方案。

我判断“香港服务器 + 域名 + HTTPS”是当前最合适的方向，原因很直接：

- 它能保留当前 `dashboard`，不需要推倒重做。
- 它能规避“中国内地公网网站/应用入口”的备案阻塞。
- 它允许继续沿用当前仓库的部署方式，不需要再引入第二套技术栈。
- 它对非技术运营最友好，后续大部分步骤可以继续交给 Codex / Claude Code 执行。

### 2.2 中国内地学员能不能访问香港看板

能访问，但体验不应被描述成“等同于中国内地服务器”。

更准确的说法是：

- 中国香港服务器不需要走中国内地网站备案流程。
- 中国内地用户访问中国香港服务器通常是跨境访问，延迟和稳定性一般会弱于中国内地同地域入口。

所以，香港机方案的本质是：

- 它是“免备案、可访问、最快形成正式公网入口”的方案；
- 它不是“网络质量与内地机完全等价”的方案。

### 2.3 当前看板能不能直接变成飞书原生系统

不能直接“原封不动部署成飞书原生页面”。

当前 `dashboard` 是标准 React/Vite SPA。它可以：

- 作为飞书里的网页入口被拉起；
- 作为飞书知识库首页、文档首页、或其他入口里的一个链接被打开。

但它不能在不改造的前提下，直接变成 Feishu Base 原生页面。

所以这件事要严格区分两种说法：

- “把当前看板挂进飞书里使用”：可以。
- “把当前 React 看板直接部署成飞书原生 Base 页面”：不可以，必须迁移或重做。

## 3. 官方依据

以下结论有官方材料支撑，不是推测：

### 3.1 关于备案

阿里云官方说明：

- 只有在中国内地提供互联网信息服务的网站/应用，才需要ICP备案。
- 中国香港等非中国内地地域，不要求走中国内地ICP备案。

可参考：

- [什么是ICP备案 - 阿里云](https://help.aliyun.com/zh/icp-filing/basic-icp-service/product-overview/what-is-an-icp-filing)
- [个人网站备案快速入门 - 阿里云](https://help.aliyun.com/zh/icp-filing/basic-icp-service/getting-started/quick-start-for-icp-filing-for-personal-websites)
- [为网站添加ICP备案号并链接至工信部 - 阿里云](https://help.aliyun.com/zh/dws/icp-number)

### 3.2 关于香港节点访问特性

阿里云官方长期口径是：中国香港和海外地域不需要中国内地备案，但其对中国内地用户访问属于跨境路径，网络体验与中国内地节点不同。

因此，本项目不应该把“中国香港免备案”错误理解成“对中国内地学员体验一定等于杭州节点”。

### 3.3 关于飞书原生路线

飞书官方有两类路线与本项目相关：

- Feishu Base / 多维表格应用模式
- 飞书低代码平台 / 自定义组件路线

可参考：

- [飞书多维表格能力总览](https://www.feishu.cn/content/base)
- [飞书低代码平台手册精选](https://www.feishu.cn/content/article/7576945446883445979)

这再次说明：飞书里可以承接“原生入口”和“低代码能力”，但并不等于当前 React 看板可以零改造直接变成 Base 原生页。

## 4. 方案对比

### 方案 A：继续使用当前杭州裸 IP + 3000

结论：不推荐作为正式公网入口。

原因：

- 入口不正规，仍是裸 IP + 端口。
- HTTPS 不可用。
- 不适合扩散给学员和运营人员长期使用。
- 容易把一期正式交付面误导成“公网独立站”，与项目当前冻结口径冲突。

### 方案 B：把前端放香港，后端继续留杭州

结论：不推荐。

原因：

- 当前前端是同源 API 模型。
- 一旦拆成“两地两端”，就会引入跨域、反向代理、路由重写和额外排障成本。
- 对非技术运营非常不友好，也不利于 Codex / Claude Code 稳定代做。

### 方案 C：整套应用迁移到阿里云中国香港 SWAS

结论：推荐。

原因：

- 不改当前应用架构。
- 保留现有 `dashboard`。
- 只新增“香港主机 + 域名 + HTTPS”这一层。
- 仍可继续使用当前的 Aliyun CLI、MCP、部署脚本和操作习惯。

### 方案 D：直接放弃公网看板，只保留飞书 Base / 知识库入口

结论：这是“最便宜、最快上线、最稳”的保守方案。

如果你不坚持保住当前 React 看板，那么这其实是最省钱的路径：

- 不需要再买香港机。
- 不需要再买新域名。
- 不需要处理 HTTPS 和公网入口。
- 一期正式入口完全落在“飞书知识库 / 文档首页 + Feishu Base 仪表盘”。

它的代价只是：当前 `dashboard` 退回工程辅助面，不再作为正式入口。

## 5. 建议的最终方案

如果目标是“两条都要”：

- 既要保住当前 React 看板；
- 又要尽快形成一个可给学员/运营使用的正式公网入口；

那么我建议采用下面这条组合方案。

### 5.1 正式建议

采用：

- 阿里云中国香港 SWAS
- 独立域名
- 域名上的 HTTPS
- 整套应用同机部署

不要采用：

- 裸 IP 正式上线
- 香港前端 + 杭州后端拆分
- 直接把 React 看板当成 Feishu Base 原生页

### 5.2 为什么推荐阿里云中国香港，而不是换别的云

不是因为别的云不行，而是因为现在没有必要引入新的复杂度。

当前项目已经围绕阿里云形成了以下资产：

- 本地 CLI
- MCP 能力
- 现有脚本
- 现有运维路径
- 你已经熟悉的服务器供应商

所以从“最快、最便宜、最容易让 AI 代做”的角度，继续留在阿里云体系内最合理。

## 6. 这件事里，Codex / Claude Code 能替你做什么

在这个方案下，AI 可以承担大部分执行工作，但不是全部。

### 6.1 AI 可以替你执行的部分

- 整理和生成部署脚本
- 调整环境变量和配置模板
- 部署整套应用到香港服务器
- 验证 `/dashboard/`、`/api/health`、`/api/feishu/status`
- 切换 `FEISHU_LEARNER_HOME_URL`
- 切换 `FEISHU_OPERATOR_HOME_URL`
- 切换 `FEISHU_LEADERBOARD_URL`
- 写回滚文档
- 做上线前后的 smoke test

### 6.2 仍然需要你人工完成的部分

- 购买香港服务器
- 购买域名
- 配置域名 DNS
- 确认账号内的计费和资源购买动作

这不是因为技术上一定完全做不到，而是因为这些动作通常直接涉及计费、购买和账号资源创建，应该保留在人工确认之下。

## 7. 如果你要“最快且兼顾价格”

我给出的排序是：

### 第一选择：香港 SWAS + 域名 + HTTPS

适用情况：

- 你要保留当前 `dashboard`
- 你希望尽快形成正式公网入口
- 你接受增加一个香港主机和一个域名成本

这是“保留看板”前提下的最优解。

### 第二选择：不迁移公网看板，直接收敛到飞书 Base / 知识库

适用情况：

- 你最在意成本
- 你最在意尽快稳定上线
- 你不要求把当前 `dashboard` 作为正式入口

这是“整体成本最低、速度最快”的解。

## 8. 我的最终结论

一句话结论：

当前杭州裸 IP + 3000 的看板入口，不适合作为正式公网入口继续扩散；如果你要保住当前 React 看板，最合理的方案是“阿里云中国香港 SWAS + 域名 + HTTPS + 整套同机部署”；如果你最在意成本和上线速度，那么最稳的做法仍然是把一期正式入口收敛到“飞书知识库 / 文档首页 + Feishu Base”。

更具体地说：

- “能不能迁移到香港”：能。
- “中国内地学员能不能访问”：能，但不要承诺体验等同于内地机。
- “是不是就不用备案了”：按中国内地ICP备案口径，可以避开中国内地备案要求。
- “当前看板能不能直接变成飞书原生页面”：不能直接，需要迁移或重做。
- “最佳综合解”：阿里云中国香港 SWAS + 域名 + HTTPS，同机承载当前整套应用。

## 9. 后续建议

如果下一步继续推进实施，我建议后续动作按这个顺序进行：

1. 先人工购买香港 SWAS 和域名。
2. 再由 Codex / Claude Code 负责部署脚本、HTTPS、环境变量和 smoke test。
3. 香港域名稳定后，再把三个 `FEISHU_*_URL` 切过去。
4. 杭州机保留 7 到 14 天作为回滚位。
5. 回滚观察期结束后，再决定是否关闭当前杭州公网看板入口。
